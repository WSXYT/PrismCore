# PrismCore 开发与贡献指南

本文档说明两件事：
- 如何使用当前仓库的开发/预发布/正式发布流程
- 如何累计贡献值，以及贡献值对应的贡献认可方式

## 1. 分支与发布流程

当前采用三分支模型：
- `dev`：日常开发分支
- `beta`：预发布分支（自动发布预发布版本）
- `master`：正式发布分支（自动发布正式版本）

推荐代码流向：
1. 在 `dev` 开发并提交
2. 通过 PR 合并 `dev -> beta`，触发预发布
3. 验证通过后通过 PR 合并 `beta -> master`，触发正式发布

## 2. 你需要维护的文件

版本和更新说明都来自 [`VERSION.md`](./VERSION.md)。

固定格式如下：

```md
# PrismCore 版本信息

Version: 0.0.4

## 待发布

### 新增
- 新增：xxx

### 变更
- 变更：xxx

### 修复
- 修复：xxx

### 其他
- 其他：xxx
```

注意事项：
- 仅使用 `新增 / 变更 / 修复 / 其他` 四个分类
- 每条更新必须单行，以 `- ` 开头
- 尽量追加新条目，不要频繁改历史条目文本

## 3. 自动发布行为说明

### 3.1 推送到 dev
- 只执行 CI 构建检查（x64 / x86 / ARM64 三平台）
- 不发布 Release

### 3.2 推送到 beta
- 自动读取 `VERSION.md` 中的 `Version: X.Y.Z`
- 自动计算并创建预发布标签：`vX.Y.Z-beta.N`
- 自动创建 GitHub Pre-release
- 预发布说明只包含相对上一个 `beta` 版本的新增条目（不重复历史条目）

### 3.3 推送到 master
- 自动读取 `VERSION.md` 中的 `Version: X.Y.Z`
- 自动创建正式标签：`vX.Y.Z`
- 自动创建正式 Release（Latest）
- 发布成功后，自动从 `dev` 的 `VERSION.md` 中删除本次已发布条目
  - 只删除已发布条目
  - 不会整文件重置
  - `dev` 中新写的未发布条目会保留

## 4. 贡献值规则

贡献值用于在项目内进行贡献记录、鸣谢和维护权限评估。

### 4.1 计分范围
- 只统计已合并到仓库默认开发流程（`dev/beta/master`）的内容
- 关闭未合并 PR、草稿实现、纯聊天建议不计分

### 4.2 计分标准

| 类型 | 分值 | 说明 |
|------|------|------|
| 功能开发（Feature） | +8 | 新功能、可见能力增强 |
| 缺陷修复（Bugfix） | +5 | 明确修复线上或测试问题 |
| 性能/稳定性优化 | +4 | 有可验证收益 |
| 测试补充/CI 改进 | +3 | 增强可回归性与质量保障 |
| 文档改进 | +2 | README、指南、注释等有效改进 |
| 重构（无行为变化） | +2 | 明显提升可维护性 |
| 发布协助（版本说明、发布验证） | +2 | 降低发布风险 |
| 回滚或修复由本人引入的问题 | -3 | 同一问题重复发生可叠加 |

### 4.3 建议的认定方式
1. 以 PR 为单位认定一次贡献值
2. 若一个 PR 涉及多类内容，按“主要贡献类型”计分一次
3. 存在争议时由维护者最终裁定

### 4.4 贡献值用途（建议）
- 累计达到 20 分：可加入 README 贡献者名单（由维护者确认）
- 累计达到 50 分：可申请协作维护权限（由仓库 owner 决定）

## 5. 推荐提交流程

1. 从 `dev` 拉取最新代码
2. 完成功能并更新 `VERSION.md`
3. 提交 PR 到 `dev`
4. 周期性合并 `dev -> beta` 进行预发布验证
5. 稳定后合并 `beta -> master` 完成正式发布

## 6. 常见问题

### Q1：为什么我看到两个 CI？
同一提交如果被分别推到两个分支（例如先 `master` 再 `dev`），会各触发一次 CI，这是正常行为。

### Q2：预发布会不会变成 Latest？
不会。`beta` 版本为 Pre-release，正式版发布后才是 Latest。

### Q3：什么时候改 `Version`？
进入新的正式发布周期前，在 `dev` 的 `VERSION.md` 中把 `Version` 改为下一个版本号。
