# PrismCore

智能 PC 优化与资源调度系统。通过 Windows 原生 API（ntdll / kernel32 / ETW / PDH）实现内存管理、CPU 调度、磁盘清理和驱动延迟诊断。

> 需要管理员权限运行。程序启动时会自动通过 UAC 提权，并通过 Mutex 确保单实例。

## 特性

- **智能内存管理** — 通过 `NtSetSystemInformation` 清理备用列表、`EmptyWorkingSet` 修剪工作集、智能识别空闲进程并将其页出到虚拟内存，三管齐下释放物理内存
- **动态虚拟内存** — 实时监控提交费用比，超限时通过 `NtCreatePagingFile` 动态扩展分页文件（立即生效，无需重启），自动选择最优分区，恢复后自动撤回
- **ProBalance CPU 调度** — 参考 Process Lasso，通过 EWMA 趋势预测 + Z-score 异常检测实现预测性干预，支持 Intel 12 代+ 混合架构 P/E-Core 智能绑定
- **DPC/ISR 延迟诊断** — PDH 性能计数器 + ETW 事件追踪双通道，精确到单个驱动的延迟归因，内置 13 种常见高延迟驱动模式识别
- **深度磁盘清理** — 临时文件、Electron 缓存、旧驱动、孤立注册表项、Windows Update 缓存、CompactOS 压缩，系统级操作前自动创建还原点
- **后台自动优化** — 压力触发式（全屏检测 + 页错误增量），无需手动干预，全程保护前台窗口和音频进程
- **Fluent Design UI** — 基于 PySide6 + QFluentWidgets，Windows 11 风格，四页导航 + 设置页

## 功能详解

应用采用四页导航：**首页** / **清理** / **加速** / **工具箱**，外加底部设置页。

### 首页 — 系统概览与智能优化

#### 健康评分

首页中央的圆形仪表盘实时显示系统健康分数（0-100），由三个维度加权计算：

| 维度 | 满分 | 满分条件 | 扣分区间 |
|------|------|----------|----------|
| 内存 | 40 | 使用率 < 70% | 70-95% 线性扣分 |
| CPU | 30 | 使用率 < 50% | 50-95% 线性扣分 |
| 磁盘 | 30 | 使用率 < 70% | 70-95% 线性扣分 |

评分 ≥ 80 显示"良好"，60-79 显示"一般，建议优化"，< 60 显示"较差，请立即优化"。

#### 一键智能优化

点击"智能优化"按钮后依次执行：

1. **清理备用列表** — 调用 `NtSetSystemInformation` 清空系统文件缓存（Standby List）。Windows 会把读过的文件数据缓存在备用列表中，长时间运行后可能占用数 GB，清理后立即释放为可用内存
2. **修剪后台工作集** — 遍历所有后台进程，调用 `EmptyWorkingSet` 将其物理内存页标记为可回收。自动跳过前台窗口、全屏应用、音频进程（audiodg / spotify / vlc / foobar2000 等）和系统关键进程（csrss / lsass / svchost 等）
3. **智能进程分页** — 识别空闲大内存进程（CPU ≈ 0%、物理内存 > 10MB、非保护进程），通过 `EmptyWorkingSet` 将其工作集页出到虚拟内存（pagefile），释放物理内存给活跃程序使用
4. **快速垃圾清理** — 清理系统临时文件和 Electron 应用缓存
5. **动态扩展分页文件** — 如果提交费用占比超过阈值（默认 70%），计算超出量 + 20% 余量（最小 256MB，累计上限 8GB），优先通过 `NtCreatePagingFile` 动态扩展（立即生效），失败时回退到 wmic（需重启）。自动选择非系统盘中可用空间最大的分区
6. **显示结果** — 展示释放的内存量和优化前后的评分变化

> 扩展分页文件后会弹出带"撤回"按钮的通知，点击可删除临时分页文件配置。临时分页文件活跃超过 30 分钟时，会弹出"建议转为永久设置"的智能建议。

#### 实时监控

每 1.5 秒自动采集一次：
- CPU 使用率
- 内存使用率（已用 / 总量）
- DPC/ISR 延迟占比（通过 PDH 性能计数器），高延迟时自动识别问题驱动并给出建议
- 各磁盘使用率和可用空间

#### 后台自动优化

开启后（设置页配置），程序在后台定时检测系统压力，满足条件时自动执行优化：

**触发条件（压力触发式）：**
- 可用内存 < 2GB 且备用列表 > 总内存 25%
- 同时满足以下任一：检测到全屏应用（通过 `GetWindowRect` 对比屏幕分辨率）或页错误增量超过阈值（50000）

**自动执行的操作：**
- 清理备用列表 + 修剪后台工作集 + 空闲进程分页
- 提交费用比 ≥ 阈值时自动扩展分页文件
- 提交比降至阈值 - 15% 以下时自动撤回临时分页文件
- 连续 3 次检测超限时弹出"建议设置虚拟内存"通知，附带"一键设置"按钮

优化动作会在首页右下角弹出通知，不会打断用户操作。

### 清理页 — 磁盘垃圾清理

#### 扫描模式

- **快速模式** — 跳过大文件扫描，速度更快，适合日常使用
- **深度模式** — 额外扫描 C 盘 > 500MB 的大文件，帮助找出占用空间的"隐形大户"

#### 扫描范围

| 类别 | 说明 | 原理 |
|------|------|------|
| 系统临时文件 | .tmp / .log / .old / .bak / .dmp / .etl / .chk | 扫描系统 TEMP 目录，匹配已知临时文件扩展名 |
| Electron 应用缓存 | Cache / GPUCache / Code Cache / blob_storage / Service Worker | 微信、QQ、VS Code、Discord 等基于 Electron 的应用会产生大量缓存，删除后自动重建 |
| 回收站 | 查询回收站占用空间 | 回收站中未彻底删除的文件仍占用磁盘 |
| Windows Update 缓存 | SoftwareDistribution\Download 目录 | 更新安装完成后下载包不再需要 |
| 旧驱动 | DriverStore 中同类驱动的旧版本 | Windows 保留所有历史版本驱动，只需保留最新版 |
| 孤立注册表项 | CLSID / AppPath / Uninstall 中引用已不存在的文件 | 软件卸载后残留的注册表指向，清理前自动备份到 `%LOCALAPPDATA%\PrismCore\RegBackup` |
| CompactOS | 检测系统是否已启用 NTFS 压缩 | 未压缩时显示可压缩选项，可节省数 GB 磁盘空间 |
| 大文件（深度模式） | C 盘 > 500MB 的文件 | 通过 MFT 快速扫描，帮助定位大文件 |

#### 安全机制

系统级清理（旧驱动、注册表、WinSxS 组件存储、更新缓存、CompactOS）执行前会自动创建 Windows 系统还原点，出问题可通过"系统还原"恢复。注册表清理额外备份到本地目录。

### 加速页 — 内存优化与 CPU 调度

加速页包含三个功能区域：内存优化、启动项管理、进程管理。

#### 内存优化

顶部卡片显示当前内存使用情况和提交费用比率。点击"一键优化"执行与首页相同的智能优化流程（清理备用列表 → 修剪工作集 → 空闲进程分页 → 动态扩展分页文件）。

**动态虚拟内存热调整原理：**

Windows 的"提交费用"（Commit Charge）是所有程序申请的虚拟内存总量，"提交限制"（Commit Limit）= 物理内存 + 分页文件大小。当提交费用接近提交限制时，新程序无法分配内存，系统会变得不稳定。

PrismCore 的应对策略：
1. 实时监控提交费用 / 提交限制的比率
2. 比率超过扩展阈值（默认 70%）时触发智能线性扩展
3. 计算公式：扩展量 = (实际提交费用 - 阈值对应的提交量) × 1.2，最小 256MB
4. 优先调用 `NtCreatePagingFile` 在非系统盘动态创建分页文件（立即生效，无需重启）
5. `NtCreatePagingFile` 失败时回退到 wmic 方式（需重启生效）
6. 累计扩展上限 8GB，防止磁盘被过度占用
7. 自动选择非系统盘中可用空间最大且 > 8GB 的分区，无合适分区时回退到 C 盘

#### ProBalance CPU 调度

参考 Process Lasso 的 ProBalance 机制，自动检测 CPU 霸占进程并临时约束其优先级，防止单个程序拖慢整个系统。

**工作原理：**

1. **EWMA 趋势预测** — 对系统总 CPU 和每个进程的 CPU 使用率进行指数加权移动平均（Exponential Weighted Moving Average），计算变化率并向前预测 2 步。当预测值即将超过阈值时提前干预，而不是等到已经卡顿才反应
2. **Z-score 异常检测** — 跟踪每个进程的 CPU 使用率历史基线，当某个进程的 CPU 突然偏离基线超过设定的标准差倍数（默认 2.5σ）时，即使绝对值未达阈值也会触发约束。这能捕捉到"平时安静但突然暴走"的进程
3. **约束动作** — 将目标进程优先级降为 `BELOW_NORMAL`；在 Intel 12 代+ 混合架构 CPU 上，额外通过 `SetProcessAffinityMask` 将进程绑定到 E-Core（能效核），把 P-Core（性能核）留给重要程序
4. **自动还原** — 系统 CPU 降至恢复阈值（默认 40%）以下时，自动还原所有被约束进程的原始优先级和 CPU 亲和性。设有最短约束时间（3 秒）防止优先级抖动
5. **白名单保护** — 系统关键进程、音频进程、前台窗口进程、桌面管理器（dwm.exe / explorer.exe）等绝对不会被约束

**P/E-Core 拓扑检测：** 通过 `GetSystemCpuSetInformation` API 精确识别 CPU 核心类型，非混合架构（如 AMD）仅调整优先级，不绑定核心。

**手动调度：** 在进程列表中也可以手动操作：
- **提升** — 将进程设为高优先级并绑定 P-Core，适合游戏、视频编辑等需要高性能的程序
- **降低** — 将进程设为空闲优先级并绑定 E-Core，适合后台更新、同步工具等不重要的程序

#### 启动项管理

列出注册表（`HKCU\...\Run`、`HKLM\...\Run`）和计划任务中的所有开机自启动项。每个启动项旁边有开关，可以直接启用或禁用。禁用启动项不会卸载软件，只是让它不再开机自动运行，仍可手动打开。启动项越少，开机速度越快。

#### 进程管理

切换到加速页时自动加载，按内存占用从高到低排序显示当前运行的进程。可以对单个进程执行提升/降低优先级操作（原理同上述 ProBalance 手动调度）。

### DPC/ISR 延迟诊断

DPC（Deferred Procedure Call）和 ISR（Interrupt Service Routine）是 Windows 内核中驱动程序处理硬件中断的机制。当某个驱动的 DPC/ISR 执行时间过长时，会导致音频爆音、视频卡顿、鼠标不流畅等问题。

**双通道检测方案：**

| 通道 | 技术 | 作用 |
|------|------|------|
| PDH | 性能计数器 `\Processor(_Total)\% DPC Time` 等 | 提供可靠的总体 DPC/ISR 时间百分比，判定是否超阈值 |
| ETW | 内核事件追踪（Event Tracing for Windows） | 捕获每个 DPC/ISR 事件的回调地址，精确归因到具体驱动 |

**工作流程：**
1. PDH 每 1.5 秒采集一次 DPC 时间占比（阈值 3%）和 ISR 时间占比（阈值 2%）
2. 超阈值时，ETW 提供的驱动级统计定位问题根源
3. ETW 通过 `EnumDeviceDrivers` 获取所有内核模块的基地址，用二分查找将 DPC/ISR 回调地址映射到具体驱动模块
4. 匹配内置的 13 种已知高延迟驱动模式，给出具体建议

**已知高延迟驱动模式：**

| 驱动模块 | 来源 | 建议 |
|----------|------|------|
| nvlddmkm | NVIDIA 显卡 | 更新驱动或关闭后台录制（ShadowPlay） |
| atikmdag | AMD 显卡 | 更新驱动 |
| igdkmd | Intel 核显 | 更新驱动 |
| rtwlane | Realtek 无线网卡 | 更新驱动或禁用节能模式 |
| rt640x64 | Realtek 有线网卡 | 更新驱动 |
| rtkvhd64 | Realtek 音频 | 更新驱动 |
| hdaudbus | HD Audio 总线 | 检查音频驱动 |
| ndis | 网络驱动接口 | 检查网卡驱动 |
| tcpip | TCP/IP 协议栈 | 检查网络负载 |
| storport | 存储端口 | 检查磁盘健康 |
| stornvme | NVMe 存储 | 更新固件 |
| usbxhci | USB 3.0 控制器 | 检查 USB 设备 |
| bthhfenum | 蓝牙 | 更新蓝牙驱动 |

## 设置详解

设置页位于左侧导航栏底部，分为六个区域。所有设置通过 QSettings 持久化，修改后立即生效。

### 后台自动优化

#### 启用后台自动优化
- **默认**：开启
- **作用**：开启后程序在后台定时检测内存状态，满足压力条件时自动执行清理备用列表和修剪工作集
- **建议**：保持开启。后台检测非常轻量（仅读取内存状态），只在真正需要时才执行优化操作，不会影响日常使用

#### 检测间隔
- **范围**：15 - 120 秒
- **默认**：15 秒
- **作用**：后台自动优化的检测周期。每隔这么多秒检查一次系统内存压力
- **建议**：日常使用保持默认 15 秒即可。如果觉得通知太频繁，可以调大到 30-60 秒。玩游戏时建议保持较短间隔以便及时响应内存压力

#### 内存阈值
- **范围**：50% - 95%
- **默认**：70%
- **作用**：内存使用率超过此值时才会触发自动优化。低于此值时即使定时器到期也不会执行任何操作
- **建议**：8GB 内存的电脑建议设为 70-80%，16GB 及以上可以设为 80-85%。设得太低会导致频繁优化，设得太高则可能来不及响应

### 虚拟内存

#### 自动创建临时分页文件
- **默认**：开启
- **作用**：智能优化时，如果清理备用列表和修剪工作集后内存仍然紧张（提交费用比超过扩展阈值），自动在非系统盘创建临时分页文件来扩展虚拟内存容量
- **建议**：保持开启。临时分页文件会在压力缓解后自动撤回，不会永久占用磁盘空间。如果你的电脑只有一个 C 盘且空间紧张，可以关闭

#### 分页文件扩展阈值
- **范围**：50% - 95%
- **默认**：70%
- **作用**：提交费用占提交限制的百分比超过此值时，触发动态扩展分页文件。提交费用 = 所有程序申请的虚拟内存总量，提交限制 = 物理内存 + 现有分页文件大小
- **建议**：默认 70% 适合大多数场景。如果经常运行大型程序（如 Chrome 开几十个标签页、大型游戏、视频编辑），可以适当调低到 60-70% 以便更早介入

#### 智能建议
- **默认**：开启
- **作用**：当临时分页文件持续活跃超过 30 分钟时，说明系统长期需要额外虚拟内存，此时会弹出通知建议将临时分页文件转为永久设置（通过 wmic 配置，需重启生效）
- **建议**：保持开启。如果你不想被建议打扰，可以关闭

### CPU 智能调度（ProBalance）

#### 启用 ProBalance
- **默认**：开启
- **作用**：自动检测 CPU 霸占进程并临时降低其优先级，系统负载恢复后自动还原。防止单个后台程序拖慢整个系统
- **建议**：保持开启。如果你运行的后台程序确实需要高 CPU（如编译、渲染），可以临时关闭

#### 系统 CPU 激活阈值
- **范围**：30% - 95%
- **默认**：60%
- **作用**：系统总 CPU 使用率超过此值时，ProBalance 才开始评估各个进程。低于此值时不会干预任何进程
- **建议**：默认 60% 适合大多数场景。如果你希望 ProBalance 更积极地介入，可以调低到 40-50%；如果觉得干预太频繁，调高到 70-80%

#### 单进程 CPU 约束阈值
- **范围**：5% - 50%
- **默认**：10%
- **作用**：单个后台进程的 CPU 使用率（EWMA 预测值）超过此值且持续数秒后，会被临时约束（降低优先级 + 绑定 E-Core）
- **建议**：默认 10% 意味着后台进程占用超过 10% CPU 就会被约束。如果你的 CPU 核心数多（8 核+），可以适当调高到 15-20%

### 高级设置（异常检测）

#### 启用 Z-score 异常检测
- **默认**：开启
- **作用**：通过统计学方法检测进程 CPU 行为突变。传统阈值检测只看绝对值（如 CPU > 10%），而 Z-score 检测的是"相对于自身历史基线的偏离程度"。一个平时 CPU 为 0% 的进程突然跳到 8%，虽然绝对值不高，但行为异常，Z-score 会捕捉到这种突变
- **建议**：保持开启。这是 ProBalance 的补充机制，能捕捉到传统阈值检测遗漏的异常进程

#### Z-score 阈值
- **范围**：1.0 - 10.0
- **默认**：2.5
- **作用**：进程 CPU 使用率偏离历史基线超过此倍标准差时触发约束。值越小越敏感，越容易触发；值越大越宽松
- **建议**：默认 2.5 表示偏离 2.5 个标准差才触发，适合大多数场景。如果误报太多（正常程序被约束），调高到 3.0-4.0；如果想更敏感，调低到 2.0

#### EWMA 平滑系数
- **范围**：0.05 - 0.95
- **默认**：0.4
- **作用**：指数加权移动平均的敏感度参数（α）。α 越大，对最近的 CPU 变化越敏感，预测值跟随实际值越紧密；α 越小，越平滑，对短暂波动不敏感
- **建议**：默认 0.4 是平衡选择。如果希望更快响应 CPU 突变，调高到 0.5-0.6；如果希望减少因短暂波动导致的误判，调低到 0.2-0.3

### 内存优化策略

这三个开关控制智能优化时具体执行哪些内存操作，可以根据需要单独开关。

#### 备用列表清理
- **默认**：开启
- **作用**：清空系统文件缓存（Standby List），将缓存的文件数据释放为可用内存。这是争议最小的优化手段，因为备用列表本质上是"可以随时丢弃的缓存"
- **建议**：保持开启。唯一的副作用是清理后首次访问某些文件会稍慢（需要重新从磁盘读取），但 SSD 上几乎感觉不到

#### 工作集修剪
- **默认**：开启
- **作用**：调用 `EmptyWorkingSet` 修剪后台进程的物理内存页，将其标记为可回收。被修剪的内存页仍在物理内存中（进入备用列表），进程再次访问时可以快速恢复
- **建议**：保持开启。有一定争议：虽然能立即释放内存，但如果被修剪的进程很快又需要这些内存页，会产生"软页错误"（从备用列表恢复）甚至"硬页错误"（从磁盘读回）。PrismCore 通过只修剪低优先级后台进程来降低这个风险。如果你发现优化后某些后台程序变慢，可以关闭此选项

#### 空闲进程分页
- **默认**：开启
- **作用**：识别空闲大内存进程（CPU ≈ 0%、物理内存 > 10MB），将其工作集页出到虚拟内存（pagefile）。比工作集修剪更激进，释放的内存更多
- **建议**：保持开启。与工作集修剪类似的争议，但只针对"确实空闲"的进程。如果你的电脑有 SSD 且虚拟内存充足，副作用很小。如果是 HDD，进程恢复时可能会有明显延迟，可以考虑关闭

### 系统监控

#### DPC/ISR 延迟监控
- **默认**：开启
- **作用**：通过 PDH 性能计数器实时监测驱动程序延迟（DPC/ISR）。高延迟会导致音频爆音、视频卡顿、鼠标不流畅。开启 ETW 驱动级归因后还能精确定位是哪个驱动导致的问题
- **建议**：保持开启。如果你不关心延迟问题或者发现 ETW 会话与其他性能分析工具冲突，可以关闭

## 架构

MVVM 模式，View 不直接调用 Model，通过 ViewModel 信号槽中转。所有耗时操作在 QThread 中执行，不阻塞 UI。Windows API 调用统一封装在 `src/utils/winapi.py` 中。

```
src/
├── models/          # 业务逻辑（内存、CPU、清理、延迟监控等 13 个模块）
├── viewmodels/      # 信号适配（dashboard / cleaner / optimizer / toolbox）
├── views/           # 界面（main_window + 5 个页面视图）
└── utils/           # 工具（constants / winapi / privilege）
```

关键模块：

| 文件 | 职责 |
|------|------|
| `models/memory.py` | 备用列表清理、工作集修剪、分页文件调整、空闲进程分页 |
| `models/auto_optimizer.py` | 后台压力触发优化、健康评分计算、临时分页文件动态创建/扩展/撤回 |
| `models/cpu_optimizer.py` | ProBalance 引擎（EWMA 预测 + Z-score 异常检测 + P/E-Core 调度） |
| `models/latency_monitor.py` | DPC/ISR 监控（PDH 总体百分比 + ETW 驱动级归因） |
| `models/anomaly.py` | Z-score 异常检测器 |
| `models/cleaner.py` | 垃圾文件扫描（临时文件、Electron 缓存） |
| `models/system_cleaner.py` | 系统级清理（旧驱动、注册表、WinSxS、CompactOS） |
| `models/mft_scanner.py` | MFT 快速大文件扫描 |
| `utils/winapi.py` | Windows API 统一封装（ntdll / kernel32 / psapi / advapi32 / pdh） |

## 常见问题

**清理后文件能恢复吗？**
基础清理删除的是缓存和临时文件，软件会自动重新生成，不影响正常使用。系统级清理（旧驱动、注册表、WinSxS 等）执行前会自动创建系统还原点，可通过 Windows 的"系统还原"功能恢复。注册表清理额外备份到 `%LOCALAPPDATA%\PrismCore\RegBackup`。

**优化后需要重启吗？**
大部分优化立即生效。只有两种情况需要重启：通过 wmic 方式调整虚拟内存（`NtCreatePagingFile` 动态扩展不需要重启），以及网络重置（Winsock / TCP/IP）。

**禁用启动项会影响软件使用吗？**
不会。禁用只是让它不再开机自动运行，你仍然可以手动打开该软件。

**后台自动优化会影响性能吗？**
不会。后台检测只是定时读取内存状态（psutil），非常轻量。只在检测到内存压力且满足触发条件时才执行优化操作。

**非混合架构 CPU（如 AMD）能用 ProBalance 吗？**
可以。ProBalance 通过 `GetSystemCpuSetInformation` 检测 CPU 拓扑，非混合架构仅调整进程优先级，不绑定核心。所有优先级调度和异常检测功能正常工作。

**NtCreatePagingFile 和 wmic 有什么区别？**
`NtCreatePagingFile` 是 ntdll 未文档化 API，可以在运行时动态创建/扩展分页文件，立即生效无需重启。wmic 是通过 WMI 修改注册表中的分页文件配置，需要重启才能生效。PrismCore 优先使用前者，失败时回退到后者。

**ETW 会话启动失败怎么办？**
ETW 内核追踪需要管理员权限和 `SeSystemProfilePrivilege` 特权。如果启动失败（可能被其他性能分析工具占用），DPC/ISR 监控会自动降级为仅使用 PDH 性能计数器，仍然可以检测总体延迟百分比，只是无法精确归因到具体驱动。
